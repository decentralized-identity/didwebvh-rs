# Spec: v0.5.1 — didwebvh 1.0 spec-compliance patch

## Objective

Land a non-breaking patch release that closes the gaps flagged in
[didwebvh-test-suite PR #4](https://github.com/decentralized-identity/didwebvh-test-suite/pull/4)
and refreshes dependencies. Consumers upgrade from 0.5.0 → 0.5.1 with no code
changes.

**Why now:** PR #4 surfaces cross-implementation compliance issues between
didwebvh-rs, didwebvh-ts (v2.7.3), and didwebvh-py. Of the three Rust items
called out in the PR, one is a real spec-vs-code bug, one is an interop parity
enhancement, and one is a false positive in the suite itself. The patch lands
the two real items plus regression fixtures.

**Success, bluntly:** after merge, the test-suite harness at
`implementations/rust/src/main.rs` needs fewer XFAILs, and three of the four
workarounds in its `normalize_actual()` become unnecessary.

## In-scope work

### 1. Relax deactivation strictness (real bug)

**File**: `src/lib.rs:371-383`.

**Current behaviour**: `create_log_entry()` rejects any deactivation entry
whose `update_keys` is not empty `[]`, with
`"Cannot deactivate DID unless update_keys is set to []"`.

**Spec** ([§Deactivate](https://identity.foundation/didwebvh/v1.0/#deactivate-revoke)):
"`updateKeys` **SHOULD** be set to an empty array `[]` when deactivating the
DID." SHOULD, not MUST.

**Fix**: drop the error. When a deactivation entry arrives with non-empty
`update_keys`, accept it. No warning log — the spec doesn't require one, and
we already treat SHOULDs as advisory elsewhere.

**Also touched**: `src/parameters/mod.rs:190-197` (the `Deactivated` branch in
`diff()` does not currently enforce the empty-update-keys rule — good, leave
it alone).

### 2. Add `versionNumber` to `MetaData` (interop parity)

**File**: `src/log_entry/mod.rs:39-58`.

**Current**: `MetaData` exposes `version_id: String` (e.g. `"2-Qm..."`) but
consumers must parse the integer themselves.

**Enhancement**: add `version_number: u32` field, derived from the same
`version_id` that already populates the struct. TS emits this; our lib should
too, purely for DX/interop parity.

**Serde**: rename to `versionNumber` on the JSON side to match TS and the
query parameter of the same name.

**Non-breaking** check: `MetaData` is `Serialize + Deserialize`. Adding a
field is **source-compatible** for struct literals only if all existing
construction sites stay inside the crate; if any public code constructs
`MetaData { ... }` directly, this becomes breaking. Verification step:
`grep -r "MetaData {" src/ examples/ tests/` — must be crate-internal only.
If external construction is possible, mark the new field `#[serde(default)]`
and add a `#[non_exhaustive]` attribute or — simpler — just document as
"public API additions are not breaking under semver".

### 3. Commit PR#4 test-suite vectors as regression fixtures

**New directory**: `tests/test_vectors/test_suite/`.

**Contents**: copy `did.jsonl` + `resolutionResult.json` from each of these
PR #4 scenarios, plus a README explaining provenance:

- basic-update, basic-create
- key-rotation
- multi-update, multiple-update-keys
- deactivate (after fix #1, this vector must validate)
- services, witness-update, witness-threshold
- portable, portable-move
- pre-rotation, pre-rotation-consume

**New test**: `tests/test_suite_interop.rs` iterates the fixtures, runs each
through `DIDWebVHState::resolve_log_owned()`, compares against the committed
`resolutionResult.json`. Minimum acceptance: each vector's log validates end-
to-end (no truncation). Deep metadata comparison can come later if needed.

### 4. Dependency updates

**Approach**: run `cargo update` within current `Cargo.toml` ranges, plus
pick up any compatible patch bumps on the direct deps that have them. **No
version-range changes in `Cargo.toml`** — that's the threshold between patch
and minor. Exceptions need an explicit line-item below; none expected.

**Verify**: `cargo build --all-features`, `cargo test --all-features`,
`cargo clippy --all-features --all-targets -- -D warnings` all pass.

## Out of scope

- **"nextKeyHashes: [] rejection"**: verified via reproducers against
  didwebvh-ts and (monkey-patched) didwebvh-py fixtures — our lib resolves
  them. The PR's XFAIL is based on a pre-run code inspection that doesn't
  reflect current behaviour. No fix needed.
- **Service ordering (#files vs #whois)**: Glenn's PR comment: "no standard;
  avoid making it consistent — risk of false assumption by consumers."
- **Resolution envelope `@context`, `didResolutionMetadata` shape, service
  ID fragment normalisation**: these are TS-vs-Python differences, not
  Rust-specific.
- **API breakage**: anything that would force a minor bump. If a fix turns
  out to require a breaking change, stop, escalate to 0.6.0 discussion.

## Testing Strategy

- Unit tests: colocated with code under test (existing pattern).
- Integration tests: `tests/` directory, one file per scenario.
- Fixtures: `tests/test_vectors/`.
- Crypto assertions use `affinidi_secrets_resolver::Secret::generate_ed25519`.
- Tokio runtime: `#[tokio::test]` with the `rt-multi-thread,macros` features.

**For each fix:**

| Item | Test |
|---|---|
| Deactivation relaxation | Test commits the PR#4 `deactivate` vector and asserts it resolves cleanly (`ValidationReport::truncated.is_none()`). Plus a new unit test that builds a log via `update_did` with deactivation+non-empty updateKeys and asserts success. |
| `versionNumber` field | Serialize a `MetaData` and assert the JSON contains `"versionNumber": <int>` matching the leading integer of `versionId`. |
| Interop regression | `tests/test_suite_interop.rs` walks `tests/test_vectors/test_suite/` and resolves each log. |

## Boundaries

**Always do:**
- Run `cargo fmt` before every commit (project rule).
- Sign every commit with DCO `-s` (project rule).
- Keep changes additive — no field removal, no signature change, no error
  type rename.
- Update `CHANGELOG.md` under a new `## 0.5.1` section for each in-scope item.

**Ask first:**
- Any change that breaks build for consumers (public signature change,
  removed pub item, bumped MSRV).
- Any `Cargo.toml` direct-dep **range** change (e.g. `affinidi-* 0.5 → 0.6`).
- Adding a new dependency.
- Any behaviour change outside the three items above.

**Never do:**
- Commit PR#4 fixtures that contain keys/secrets (the `did.jsonl` vectors
  don't, but `did-secrets.json`-style files do — exclude).
- Bump the crate version in `Cargo.toml` until every fix + tests + changelog
  are in.
- Publish to crates.io from this branch — the release flow is a separate
  follow-up.

## Resolved decisions

1. **Deactivation fix is silent** — no warning log when `update_keys` is
   non-empty alongside `deactivated: true`. Spec's SHOULD is advice to
   writers; readers accept without comment.
2. **`version_number: u32`** — matches existing `LogEntryState::version_number`.
3. **Crates.io publish in the same PR** — version bump to 0.5.1 is the last
   commit on the branch; merge triggers the existing release flow.
